# inventory/

This directory is bind-mounted into the `rhis-provisioner` container at  
`/rhis/vars/external_inventory/`.

## Setup

1. Copy `hosts.SAMPLE` â†’ `hosts` and fill in your actual hostnames and IPs.
2. The `hosts` file is referenced by all rhis-builder playbooks as:
   ```
   --inventory /rhis/vars/external_inventory/hosts
   ```

## Note

Do **not** commit plain-text passwords here.  Reference vault variables instead  
(e.g. `{{ global_admin_password }}`).  Store secrets in `~/.ansible/conf/env.yml`  
encrypted with `ansible-vault`.

In normal RHIS runs, `inventory/hosts` is auto-generated by `run_rhis_install_sequence.sh`, so `hosts.SAMPLE` mainly serves as a reference/template.

## AAP bundle installer inventories (Jinja templates)

The AAP kickstart renders installer inventories from Jinja2 templates in:

- `inventory/aap/`

Output files written inside the AAP VM:

- `/root/aap-setup/inventory`
- `/root/aap-setup/inventory-growth`

### Interactive architecture submenu

When running interactively without a pre-configured or CLI-pinned inventory template,
the script presents a guided submenu:

```
+--------------------------------------------------------------+
|       AAP Installer Inventory Architecture Selection         |
+--------------------------------------------------------------+

  0) Exit              -- Return to previous menu
  1) inventory         -- Enterprise / Multi-Node deployment
  2) About inventory   -- Name, synopsis, diagram & guidance
  3) inventory-growth  -- Growth / Single-Node containerized
  4) About inventory-growth
                       -- Name, synopsis, diagram & guidance
```

Selecting **2** or **4** shows a full "About" page for that template:
name, synopsis, ASCII architecture diagram, which templates are used,
how to set up, and why Red Hat recommends that topology.
After viewing, the submenu loops so you can still choose 1, 3, or 0.

`--DEMO` automatically bypasses the submenu and forces `DEMO-inventory.j2`
with a skip notice printed to the console.

### Available deployment models

#### 1) Enterprise / Multi-Node  (`inventory.j2`)

All AAP components run on separate dedicated VMs.  Enables independent
scaling, HA, and fault isolation for production environments.

```
                   Internet / clients
                           |
      +---------------------------------------------+
      |  [automationgateway]  (x2)                  |
      +-------------------+-------------------------+
                          |
         +----------------+--------------+
         |                |              |
 +-------+------+  +------+------+  +----+--------+
 |[automationctrl]  |[automationhub]  |[automationeda]
 |   (x2)       |  |   (x2)      |  |   (x2)     |
 +-------+------+  +-------------+  +------------+
         |
 +-------+--------------------------------------------+
 |  [execution_nodes]  (x3)  +  [redis] (x6)          |
 +----------------------------------------------------+
```

Templates:

- `AAP_INVENTORY_TEMPLATE` â†’ `inventory.j2`
- `AAP_INVENTORY_GROWTH_TEMPLATE` â†’ `inventory-growth.j2`

#### 2) Growth / Single-Node  (`inventory-growth.j2`)

All AAP components co-located on one VM using containerized services.
Recommended starting point; scale to enterprise topology when needed.

```
 +----------------------------------------------------------+
 |               aap.domain  (single VM)                   |
 |  [automationgateway]   [automationcontroller]            |
 |  [automationhub]       [automationeda]                   |
 |  [database]            redis (standalone)                |
 |  ansible_connection=local                                |
 +----------------------------------------------------------+
```

Templates:

- `AAP_INVENTORY_TEMPLATE` â†’ `inventory-growth.j2`
- `AAP_INVENTORY_GROWTH_TEMPLATE` â†’ `inventory-growth.j2`

#### 3) DEMO (`--DEMO`)

Forced automatically; no submenu shown.

```
 +------------------------------+
 | aap-26 (single node demo)    |
 +------------------------------+
```

- `AAP_INVENTORY_TEMPLATE` â†’ `DEMO-inventory.j2`  (forced)
- `AAP_INVENTORY_GROWTH_TEMPLATE` â†’ `inventory-growth.j2`

### Non-interactive / CLI overrides

Pin template names (or absolute paths) to skip the submenu entirely:

- `--inventory <template-or-path>`
- `--inventory-growth <template-or-path>`

Examples:

```bash
# Force DEMO model non-interactively
./run_rhis_install_sequence.sh --non-interactive --DEMO

# Pin enterprise topology
./run_rhis_install_sequence.sh --non-interactive \
   --inventory inventory.j2 \
   --inventory-growth inventory-growth.j2

# Pin growth topology
./run_rhis_install_sequence.sh --non-interactive \
   --inventory inventory-growth.j2 \
   --inventory-growth inventory-growth.j2
```
